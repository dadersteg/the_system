## 1. What the project is (my understanding â€” please confirm)

"The System" is a **single-user life/work automation suite** â€” a "Second Brain." The engine is **Google Apps Script (GAS)** orchestrating Google Tasks, Gmail, Drive, Calendar and Sheets, with **Google Gemini** as the reasoning layer. Around that core run **Node.js and Python daemons** (under pm2 on a Mac) that ingest chat messages (WhatsApp/Signal/iMessage/SMS via **Beeper**, Facebook **Messenger**, **Telegram**) and a set of Python sync/maintenance scripts.

Conceptually: **Google Tasks is the single execution funnel**, Drive is permanent knowledge, everything is filed under a strict 4-level taxonomy (LOS private / PMTOS work). AI "personas" (Architect, Taskmaster, Clerk, Vantage, Atlas) are prompt frameworks, not separate services.

**Structure of the working repo:**

- `src/*.js` â€” ~37 GAS modules. Heavyweights: `Code_TaskEngine.js` (AI task routing), `Code_Tasks.js` (export/log/purge, branded "Cellsion V6.1"), `Code_Dashboard.js` + `WebApp_Dashboard.html` (a public web-app dashboard, ~1,900 + ~2,760 lines), the `Code_TheClerk_*` family (email/drive/notes/photo triage), four review modules, and `Code_Config.js`/`Code_SystemCore.js` (config + shared helpers).
- `src/ingestion/` â€” Node/Python chat bridges; `src/telegram_antigravity_bridge.py` remotely drives an IDE over Chrome DevTools Protocol.
- `scripts/` â€” ~155 Python files (`utils/`, `maintenance/`); a handful are the live pm2 sync daemons, the large majority are one-off migration/probe scripts left in place.
- `tests/` â€” ~27 mixed JS/Python test and "verify" files.
- `docs/` â€” ~40 protocol/architecture markdown files, plus prompts.
- Root â€” `README.md`, `PROJECT.md`, `plan.md`, `jules.md`, `ecosystem.config.js`, `deploy*.sh`, `.github/workflows/deploy.yml`, `.clasp-*.json`, and (notably) `Fca_Database/database.sqlite`, `tasks_private.json`, `tasks_pmt.json`.

**Deployment/release:** one codebase â†’ **two GAS projects** (private `@gmail.com`, work `@playmetech.net`) via `clasp`, switched by an `ENV` script property. A GitHub Action **force-pushes to both projects on every commit to `main`, with no test gate**. An auto-commit cron (`github_sync.sh`) generates timestamped commits, and **"Jules"** (Google's autonomous coding agent) is scheduled to refactor the codebase and its PRs are **auto-merged** by `run_backfill_loop.js`.

If any of this is wrong â€” especially **which pipelines are actually live vs. idled** (the GAS engine vs. the Python `sync_tasks_combined.py` appear to overlap, and several trigger targets don't exist) â€” some severity ratings will shift. Please correct me.

## 2. Overall assessment

This is an ambitious and largely functional system, but it carries **engineering risk far above what a personal tool normally warrants**, concentrated in four areas:

1. **Live secrets exposure.** A git-tracked SQLite DB containing **live Facebook/Messenger session cookies** sits in the working tree and is **not** gitignored, in a repo that auto-pushes to GitHub; a Gemini API key is hardcoded in tracked source; the web-app's shared secrets and both deployment URLs are committed. This is the most urgent issue.
2. **Unauthenticated, owner-privileged web-app surface.** `doGet` is deployed to `ANYONE`, runs as the owner with full Gmail/Drive/Calendar/Tasks scopes, and dispatches read-any-file, write-any-doc, create-task/draft/event, and pipeline-trigger actions **with no auth check**.
3. **Irreversible, AI-driven data mutation on tight cycles.** Tasks are permanently deleted (Google Tasks has no trash) and completed-task history is wiped â€” and the two mechanisms that should prevent data loss are undermined by a **config collision** (one sheet GID serves two conflicting purposes), **stale column indices**, **missing pagination**, and a **removed quarantine safeguard whose test no longer even imports**.
4. **Release engineering.** A red test suite that nothing runs, force-deploy to two production projects on every push, and **autonomous LLM commits auto-merged with no human gate**.

Everything else (heavy duplication, timezone bugs, quota-heavy loops, dead code) is real and worth fixing, but secondary to the above.

**Severity key.** **Critical** = data loss, security exposure, or silent incorrect behavior in a live path. **Moderate** = reliability/correctness risk under realistic conditions, or significant maintainability drag. **Minor** = hygiene, cosmetic, or latent.

## 4. Data-flow & environment model (for confirmation)

- **Task lifecycle:** Clerk pipelines (email/drive/notes) extract tasks â†’ **Importer** list. `runTaskMasterEngine` (15 min) sends Importer+ToDo to Gemini (route) â†’ a second Gemini QA pass â†’ writes back: Importerâ†’ToDo, DELETEâ†’"Triage Quarantine", SPLITâ†’milestone+subtasks. State is stored in each task's `notes` field (`SYS:`/`DA:`/`Context:` lines + a `---SYSTEM_METADATA---` JSON blob with an `ai_hash` change-detection token; for Google-assigned tasks the hash goes to Script Properties). **In parallel**, Python `sync_tasks_combined.py` (2 min) routes Privateâ†”Work and writes markdown; `sheet_sync_and_maintenance.py` (15 min) logs completed tasks to Sheets and hard-deletes them. The GASâ†”Python overlap (A3) and the two independent hard-delete/wipe paths (E1/E7) are the core correctness risk.
- **Environment switch:** `ENV` script property â†’ `IS_PMT_ENV` â†’ resource-ID ternaries; separately `isPmtAccount()` â†’ naming/prompt rewriting. These can disagree (A5).
- **Release:** `src/*.js` â†’ clasp â†’ two GAS projects; GitHub Action force-pushes both on every commit; Jules auto-refactors + auto-merges (R5).
